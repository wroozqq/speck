CASTLE-FORGE

2D био-нейронный симулятор эволюции в средневековом замке
Пояснительная записка к программному комплексу

Языки и технологии: HTML5, CSS3, ES6 JavaScript, Canvas 2D API, Node.js, Express

ВВЕДЕНИЕ

Развитие современных веб-технологий расширило горизонты для создания симуляционных и игровых систем, работающих непосредственно в браузере без необходимости загрузки сторонних бинарных файлов. В последние годы особый интерес вызывают системы, сочетающие элементы геймификации с научными концепциями: биоинформатикой, теорией эволюции, генетическими алгоритмами и искусственными нейронными сетями. Симуляция эволюционного процесса в ограниченных замкнутых пространствах представляет собой не только развлекательную игровую механику, но и мощный образовательный инструмент, позволяющий визуализировать сложные математические и биологические модели в реальном времени.

Проект "Castle-Forge: 2D Medieval Evolution Simulator" представляет собой интерактивный программный комплекс, моделирующий выживание и генетическую адаптацию простейшего организма (от аморфной споры до высшего рыцаря) в опасной среде средневекового подземелья. Игрок берет на себя роль «генетического алхимика», изменяя структуру ДНК существа с помощью реальных генов (таких как RuBisCO, COX1, LDHA, SHH и др.) и наблюдая за изменением поведения, физических параметров и визуального образа подопытного объекта. Ключевой особенностью системы является наличие встроенного нейросетевого автопилота — полносвязной нейронной сети прямого распространения с генетическим алгоритмом мутации весов, способной обучаться перемещению и сбору ресурсов в лабиринте.

Актуальность разработки обусловлена возрастающей потребностью в доступных интерактивных симуляторах искусственного интеллекта и биологических алгоритмов. Использование HTML5 Canvas API в сочетании с чистым JavaScript обеспечивает высокую частоту кадров (FPS >= 60) и мгновенный доступ с любых настольных и мобильных операционных систем. Данная документация подробно описывает полный жизненный цикл проектирования, разработки, тестирования и экономического обоснования программного продукта "Castle-Forge".

1. АНАЛИТИКО-ПОСТАНОВОЧНАЯ ЧАСТЬ

1.1 Постановка задачи

Основной целью проекта является разработка законченного интерактивного браузерного приложения «Castle-Forge», которое решает следующие ключевые задачи:

  • Игровая симуляция: Моделирование двухмерного игрового мира, представляющего собой плиточный лабиринт (замковые стены, колонны, проходы, сундуки, источники энергии и враждебные стражники).
  • Генетическая эволюция: Реализация системы из 10 биологически обоснованных генов, каждый из которых математически влияет на параметры персонажа (здоровье, урон, радиус обзора, призыв помощников-нетопырей, скорость фотосинтеза).
  • Искусственный интеллект: Создание нейросетевой модели управления существом на основе сенсорных данных (лучевое сканирование стен, вычисление векторов направления к пище и врагам), а также алгоритма поиска пути в графе (BFS) с кэшированием вычислений.
  • Экономическая составляющая: Разработка алхимического магазина характеристик и премиум-магазина кристаллов для приобретения косметических скинов и временных ускорителей ДНК.

Программное обеспечение должно состоять из статического сервера на Node.js/Express и высокопроизводительного клиентского приложения, использующего объектно-ориентированную модель данных на JavaScript ES6.

1.2 Описание архитектуры системы

Архитектура программного комплекса построена по модульному принципу и адаптирована под специфику веб-приложений. На верхнем уровне взаимодействие компонентов описывается клиент-серверной моделью, где сервер выступает в роли хостинг-провайдера статических ресурсов, а клиент выполняет всю основную вычислительную работу (рендеринг, физику, ИИ). Вся игровая логика инкапсулирована в классы:

  • NeuralNetwork: Вычисляет выходные управляющие сигналы (углы поворота и тягу движения) на основе весов и смещений (biases).
  • Creature: Базовый класс для сущностей игрового мира (игрока, стражников, летучих мышей-помощников). Отвечает за физическое движение, анимацию хвостов/щупалец, отрисовку и расчет урона.
  • Particle: Система визуальных частиц (кровь при ударе, пыль при шагах, золотые искры, ДНК спирали).

Серверная часть системы минимизирована для снижения задержек и накладных расходов. Сервер должен обеспечивать стабильную передачу статических файлов (HTML, CSS, JS, аудиофайлы) клиенту под высокой нагрузкой (до 1000 одновременных запросов).

1.4 Требования к клиентской части

Клиентская часть полностью исполняется в браузере конечного пользователя. Она должна выполнять сложные математические вычисления в реальном времени, включая рейкастинг (Raycasting) и отрисовку скелетной анимации.

  • Аппаратные требования: Двухъядерный процессор с тактовой частотой от 2.0 ГГц, не менее 2 ГБ свободной оперативной памяти, видеокарта с поддержкой аппаратного ускорения Canvas 2D.
  • Программные требования: Любой современный веб-браузер с полной поддержкой стандарта ECMAScript 2020 (ES11) и выше. Тестирование гарантирует корректную работу в Google Chrome 100+, Mozilla Firefox 98+, Apple Safari 15+, Microsoft Edge 100+.
  • Требования к экрану: Поддержка отзывчивой верстки (Responsive Web Design). Адаптивный холст Canvas подстраивается под размеры контейнера, однако оптимальный игровой опыт достигается на экранах с разрешением от 1280x720 пикселей.

1.5 Функциональные требования

Система должна реализовывать следующий обязательный набор функций:

  1. Генерация лабиринта: Создание замкнутой арены размером 2000x2000 пикселей на основе координатной сетки 20x20 плиток с препятствиями (стены, пилоны).
  2. Движение и коллизии: Расчет физики скольжения и столкновения округлых тел существ со стенами лабиринта (радиус соударения, отталкивание от граней).
  3. Система генома: Применение 10 генов:
    • RuBisCO (фотосинтез энергии от факелов);
    • COX1 (регенерация HP);
    • LDHA (спринт при зажатии клавиш);
    • COL1A1 (масштабирование размеров и добавление сегментов тела);
    • ACTA1 (увеличение базовой скорости);
    • CHIT1 (процентное поглощение урона броней);
    • SHH (атака огненными снарядами);
    • OPN1LW (увеличение радиуса обзора в темноте подземелья);
    • MBP (ускорение реакции автопилота ИИ);
    • FOXP2 (создание летучих мышей-дронов, атакующих врагов).
  4. Синергетические классы: Автоматическая активация синергий (например, «Bio-Golem» при прокачке COL1A1 и CHIT1, «Dragon-Spore» при прокачке SHH и OPN1LW) с начислением уникальных бонусов.
  5. Интерфейс «Свиток Синергий» и «Магазин Кристаллов»: Интерактивные модальные окна для покупки скинов (Default, Void Fiend, Cyber-Core, Golden Knight) и изучения древа классов.

1.6 Нефункциональные требования

Нефункциональные требования определяют качественные характеристики программного продукта:

  • Производительность: Время обработки одного такта игрового цикла не должно превышать 16.67 мс для обеспечения стабильных 60 кадров в секунду (FPS). При увеличении числа активных объектов на экране до 100 единиц FPS не должен опускаться ниже 45 кадров в секунду.
  • Эргономика: Время на освоение базового интерфейса пользователем не должно превышать 3 минут. Управление дублируется кнопками интерфейса и горячими клавишами клавиатуры.
  • Надежность и обработка исключений: Глобальный перехватчик ошибок window.onerror должен выводить диагностические сообщения в консоль разработчика и на игровой баннер, не прерывая работу всего браузера.

1.7 Архитектурные требования

Проект должен быть выполнен на чистом JavaScript (Vanilla JS) для клиентской части без использования тяжеловесных фреймворков (React, Angular, Vue), что гарантирует сверхбыструю инициализацию страницы. Файловая структура должна четко разделять серверный скрипт запуска (server.js), HTML разметку (index.html), стили оформления (style.css) и клиентскую логику (game.js).

2. ТЕОРЕТИКО-ПРОЕКТНАЯ ЧАСТЬ

2.1 Назначение игры

Игра "Castle-Forge" разработана как кроссплатформенный научно-популярный симулятор. Она направлена на визуализацию концепций адаптивного искусственного интеллекта и генетического наследования в интерактивной развлекательной форме. Симулятор позволяет наглядно исследовать, как простые математические правила (веса нейронной сети, радиусы обзора, случайные мутации) приводят к сложному эмерджентному поведению живых агентов в условиях ограниченных ресурсов и агрессивной среды.

2.2 Архитектура проекта

Игровое приложение опирается на непрерывный цикл (Game Loop), реализованный через вызовы requestAnimationFrame. На каждом шаге цикла происходит вычисление дельты времени (delta time) для сглаживания скорости симуляции на мониторах с разной частотой обновления (60 Гц, 144 Гц и более). Уравнение обновления положения объекта с учетом коэффициента сопротивления среды \(f\) имеет вид:

\(v_{t} = (v_{t-1} + a \cdot \Delta t) \cdot (1 - f \cdot \Delta t)\) (2.1)

Нейросетевой модуль автопилота представляет собой двухслойный перцептрон. На вход сети подается 10 нормированных сигналов (сенсоры препятствий, положение цели, положение угрозы, уровни сытости и здоровья). Скрытый слой использует гиперболический тангенс в качестве функции активации:

\(\tanh(x) = \frac{e^x - e^{-x}}{e^x + e^{-x}}\) (2.2)

2.4 Безопасность и Оптимизация

В процессе проектирования особое внимание уделено оптимизации отрисовки. Canvas 2D API является однопоточным, поэтому избыточные вызовы контекста (такие как частые смены цвета заполнения fillStyle, сложные пути beginPath) могут приводить к потере кадров. Для минимизации задержек были реализованы следующие решения:

  • Кэширование путей BFS: Алгоритм поиска пути в ширину (BFS) запускается не каждый кадр, а раз в 30 кадров (примерно 500 мс). Если у существа активен ген миелинизации MBP, частота обновления возрастает, обеспечивая идеальный баланс между нагрузкой на CPU и маневренностью.
  • Ограничение пула частиц: Для предотвращения утечек памяти максимальный размер массива частиц particles ограничен 150 объектами. Свежие частицы вытесняют старые по принципу циклической очереди.
  • Реактивная защита от читерства: Доступ к критическим переменным (золото, ДНК, кристаллы) закрыт через прокси-геттеры и сеттеры `Object.defineProperty`. Любая несанкционированная попытка записи аномального значения отсекается, а на экране генерируется визуальный эффект вспышки.

2.5 Интерфейс пользователя

Интерфейс пользователя спроектирован с учетом современных требований к адаптивным веб-интерфейсам. Основная разметка использует CSS Grid Layout для разделения экрана на три логических блока:

  1. Левая телеметрическая панель: Отображает шкалы здоровья (HP), сытости (массы) и прогресса эволюции, подробные боевые параметры, список открытых синергий и купленных реликвий.
  2. Центральная игровая зона: Содержит холст Canvas для рендеринга игрового мира, а также панель быстрого управления (кнопка автопилота, скролл синергий, переключатель звука).
  3. Интерактивные модальные оверлеи: Всплывающие окна с полупрозрачным размытием заднего фона (backdrop-filter: blur(8px)) для премиум-магазина скинов и алхимической лавки.

2.6 Тестирование

Концепция тестирования включает проверку корректности работы ИИ в тупиковых ветках лабиринта. На этапе проектирования закладывается тестовый полигон с искусственными преградами, где проверяется способность BFS находить обходной путь за время, не превышающее 2 мс. Кроссбраузерное тестирование включает проверку рендеринга Canvas на мобильных браузерах с разной плотностью пикселей (Retina-экраны, коэффициент масштабирования devicePixelRatio).

5. ОХРАНА ТРУДА И ТЕХНИКА БЕЗОПАСНОСТИ

5.1 Организация безопасного рабочего места разработчика

Работа разработчика программного обеспечения Castle-Forge сопряжена с длительным пребыванием в сидячем положении перед экраном видеодисплейного терминала (ВДТ). В соответствии с санитарно-эпидемиологическими правилами СП 2.2.3670-20 рабочее место должно быть организовано по эргономическим стандартам:

  • Высота рабочей поверхности стола должна устанавливаться в пределах 725–750 мм. Пространство для ног под столом должно иметь высоту не менее 600 мм и ширину не менее 500 мм.
  • Рабочий стул должен быть подъемно-поворотным, регулируемым по высоте и углам наклона сиденья и спинки.
  • Монитор компьютера располагается на расстоянии 500-700 мм от глаз пользователя, при этом верхняя кромка экрана должна находиться на уровне глаз или чуть ниже для предотвращения перенапряжения шейного отдела позвоночника.

5.2 Требования электробезопасности

Поскольку ПЭВМ относится к электроустановкам до 1000 В, рабочее помещение должно соответствовать требованиям Правил устройства электроустановок (ПУЭ):

  • Все металлические корпуса системных блоков и периферийных устройств подлежат обязательному занулению/заземлению через трехполюсные вилки с заземляющим контактом.
  • Запрещается использовать розетки и выключатели с поврежденными корпусами, оголенные провода или временную проводку («набросы»).
  • При возникновении запаха гари, нетипичного шума или искрения в оборудовании разработчик обязан немедленно обесточить рабочее место путем отключения автоматического выключателя на распределительном щите.

5.3 Пожарная безопасность

Помещение для разработки ПО по пожарной опасности относится к категории В (пожароопасные помещения). Основными источниками возгорания являются перегрузки в электрических сетях и короткие замыкания в блоках питания. В помещении должны соблюдаться следующие правила:

  1. Запрещается загромождать эвакуационные пути коробками, мебелью и кабелями питания.
  2. Помещение должно быть укомплектовано первичными средствами пожаротушения: углекислотным огнетушителем типа ОУ-2 или ОУ-3, так как углекислота не повреждает токопроводящие платы ПЭВМ.
  3. В случае обнаружения очага возгорания необходимо прекратить работу, вызвать пожарную службу по телефону 101 или 112, сообщить руководству и приступить к тушению пожара с помощью огнетушителя, предварительно полностью отключив электропитание в помещении.

5.4 Меры по снижению утомляемости при работе за компьютером

Длительная работа за монитором ведет к развитию зрительного утомления (астенопии) и синдрома сухого глаза. Для профилактики профессиональных заболеваний рекомендуется соблюдать регламентированные перерывы продолжительностью 10-15 минут через каждые два часа непрерывной работы. Во время перерывов необходимо выполнять комплексы гимнастики для глаз:

  • Круговые движения глазными яблоками по часовой стрелке и против нее (по 10 раз).
  • Перевод взгляда с близко расположенного объекта (например, метка на стекле окна) на удаленный объект за окном (правило 20-20-20).
  • Легкий массаж закрытых век кончиками пальцев для стимуляции кровообращения.

Приложение А. Реализация полносвязной нейронной сети в JavaScript

class NeuralNetwork { constructor(inputSize, hiddenSize, outputSize) { this.inputSize = inputSize; this.hiddenSize = hiddenSize; this.outputSize = outputSize; // Инициализация случайных весов от -1 до 1 this.weightsIH = Array(this.hiddenSize).fill(0).map(() => Array(this.inputSize).fill(0).map(() => Math.random() * 2 - 1) ); this.weightsHO = Array(this.outputSize).fill(0).map(() => Array(this.hiddenSize).fill(0).map(() => Math.random() * 2 - 1) ); this.biasH = Array(this.hiddenSize).fill(0).map(() => Math.random() * 2 - 1); this.biasO = Array(this.outputSize).fill(0).map(() => Math.random() * 2 - 1); } predict(inputs) { // Расчет скрытого слоя с функцией активации tanh let hidden = []; for (let i = 0; i < this.hiddenSize; i++) { let sum = this.biasH[i]; for (let j = 0; j < this.inputSize; j++) { sum += inputs[j] * this.weightsIH[i][j]; } hidden[i] = Math.tanh(sum); } // Расчет выходного слоя let outputs = []; for (let i = 0; i < this.outputSize; i++) { let sum = this.biasO[i]; for (let j = 0; j < this.hiddenSize; j++) { sum += hidden[j] * this.weightsHO[i][j]; } outputs[i] = Math.tanh(sum); } return { hidden, outputs }; } }

function findBFSPath(startCol, startRow, targetCol, targetRow) { if (startCol === targetCol && startRow === targetRow) return []; let queue = [[startCol, startRow]]; let parent = {}; let visited = new Set([`${startCol},${startRow}`]); let found = false; while (queue.length > 0) { let [c, r] = queue.shift(); if (c === targetCol && r === targetRow) { found = true; break; } // Обход 4 соседей (вверх, вправо, вниз, влево) let dirs = [[0, -1], [1, 0], [0, 1], [-1, 0]]; for (let [dc, dr] of dirs) { let nc = c + dc; let nr = r + dr; let key = `${nc},${nr}`; // Проверка границ карты и отсутствия стен (MAP_DATA[y][x] === 0) if (nc >= 0 && nc < 20 && nr >= 0 && nr < 20 && MAP_DATA[nr][nc] === 0) { if (!visited.has(key)) { visited.add(key); parent[key] = `${c},${r}`; queue.push([nc, nr]); } } } } if (!found) return null; // Восстановление пути в обратном порядке let path = []; let curr = `${targetCol},${targetRow}`; while (curr !== `${startCol},${startRow}`) { path.push(curr.split(',').map(Number)); curr = parent[curr]; } return path.reverse(); }
